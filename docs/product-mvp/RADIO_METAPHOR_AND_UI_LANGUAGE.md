# Radio Metaphor and UI Language

## Purpose
BitPod uses a radio metaphor to make a complex system feel intuitive without sacrificing auditability.

## The core metaphor
BitPod is a **radio tower** broadcasting the Baseline, while users tune **Decks**—measured over time by SNR under real Channel Conditions.

## Signal is always positive ✅
- **Signal** = useful information.
- **Noise** = instability, overfitting, uncalibrated confidence.

## Movement language (rank changes)
- **gain / fade** for small moves
- **spike / drop** for large moves over short block time  
Avoid “dip” in default UI.

## Deviation and disclosure
Every shared output must disclose:
- **Deviation from Baseline** (%)
- **Personalization Delta** (what changed)

## Time language (block-height first)
Track record uses **block height** as canonical time.

## UX coherence guardrails
1) Keep the metaphor consistent.  
2) Signal is always positive.  
3) Always disclose deviation from baseline.  
4) Avoid slang in default UI.  
5) Detection flags; verification confirms; baseline computes.
